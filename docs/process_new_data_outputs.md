# Outputs of `process_new_data.py`

This document describes in detail all the files and artifacts produced by the two main functions in
`bertrend/bertrend_apps/prospective_demo/process_new_data.py`:

- [`train_new_model_for_period`](#train_new_model_for_period)
- [`regenerate_models`](#regenerate_models)

---

## Base Output Directory

All outputs are stored under the user's model directory:

```
{BERTREND_BASE_DIR}/models/users/{user_name}/{model_id}/
```

Where `BERTREND_BASE_DIR` defaults to `~/.bertrend` (configurable via the `.env` file).

---

## `train_new_model_for_period`

This function trains a single BERTopic model for one time period of new data, merges it with previously saved models,
classifies signals, and generates LLM-based interpretations.

### 1. BERTrend Model File

**Path:** `{models_path}/bertrend.dill`

The entire `BERTrend` object is serialized using `dill` (an enhanced version of `pickle` for complex objects). This file
contains:

- All trained topic models and their merge histories
- Document groups (`doc_groups`) and embedding groups (`emb_groups`) per timestamp
- The last trained BERTopic model and its timestamp
- Signal popularity data and configuration

This file is overwritten each time a new period is processed, always reflecting the latest cumulative state.

### 2. Per-Period BERTopic Model Directory

**Path:** `{models_path}/{YYYY-MM-DD}/`

For each time period (identified by `reference_timestamp`), a subdirectory is created containing:

| File                 | Format        | Description                                                                                                                                        |
|----------------------|---------------|----------------------------------------------------------------------------------------------------------------------------------------------------|
| BERTopic model files | `safetensors` | The BERTopic model serialized using the safetensors format (without c-TF-IDF, with embedding model reference)                                      |
| `doc_info_df.pkl`    | Pickle        | DataFrame with per-document information: document text, assigned topic, probability, paragraph text, source, and URL                               |
| `topic_info_df.pkl`  | Pickle        | DataFrame with per-topic information: topic ID, word count, topic name/label, representation keywords, representative documents, sources, and URLs |

### 3. Signal Classification Parquet Files

**Path:** `{models_path}/interpretation/{YYYY-MM-DD}/{signal_type}.parquet`

After training and merging, the function classifies all topics into three signal categories based on popularity
quartiles (Q1/Q3 thresholds). For each non-empty category, a Parquet file is saved:

| File Name                | Signal Type    | Description                                                                     |
|--------------------------|----------------|---------------------------------------------------------------------------------|
| `noise.parquet`          | Noise          | Topics with low and non-rising popularity (below Q1 threshold)                  |
| `weak_signals.parquet`   | Weak Signals   | Topics with low but rising popularity, or moderate popularity between Q1 and Q3 |
| `strong_signals.parquet` | Strong Signals | Topics with high popularity (above Q3 threshold)                                |

Each Parquet file is a DataFrame containing the following columns:

| Column              | Type      | Description                                                            |
|---------------------|-----------|------------------------------------------------------------------------|
| `Topic`             | int       | Topic identifier                                                       |
| `Count`             | int       | Number of documents in the topic                                       |
| `Representation`    | list[str] | Top keywords representing the topic                                    |
| `Documents`         | list[str] | Representative documents for the topic                                 |
| `Latest_Popularity` | float     | Most recent popularity score                                           |
| `LLM Title`         | str       | Short title generated by an LLM based on topic keywords and documents  |
| `LLM Description`   | str       | Detailed description generated by an LLM summarizing the topic         |
| `URLs`              | list[str] | Deduplicated list of source URLs associated with the topic's documents |

### 4. LLM Interpretation JSONL Files

**Path:** `{models_path}/interpretation/{YYYY-MM-DD}/{signal_type}_interpretation.jsonl`

For each signal category (noise, weak_signals, strong_signals), a JSONL file is generated containing detailed LLM-based
analysis of the **top K topics** (default K=5, sorted by descending `Latest_Popularity`).

Each line in the JSONL file is a JSON object with:

| Field      | Type       | Description                                                                                                                                               |
|------------|------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------|
| `topic`    | int        | Topic identifier                                                                                                                                          |
| `summary`  | str (JSON) | JSON-serialized `SignalSummary` Pydantic model — a structured summary of the signal produced by `analyze_signal()`                                        |
| `analysis` | str (JSON) | JSON-serialized `SignalAnalysis` Pydantic model — a detailed analysis of the signal's evolution, context, and implications produced by `analyze_signal()` |

These interpretations are generated in parallel (up to 5 concurrent workers) using LLM calls via the `analyze_signal`
function.

## `regenerate_models`

This function regenerates all models from scratch using the full historical dataset. It operates in two modes controlled
by the `with_analysis` parameter.

### Mode 1: `with_analysis=False` (Models Only)

In this mode, the function trains all topic models at once without per-period signal classification or LLM
interpretation.

**Outputs produced:**

1. **BERTrend model file:** `{models_path}/bertrend.dill` — the full serialized BERTrend object (same format as above).
2. **Per-period BERTopic model directories:** `{models_path}/{YYYY-MM-DD}/` — one directory per time period in the
   grouped data, each containing the BERTopic safetensors model, `doc_info_df.pkl`, and `topic_info_df.pkl`.

No interpretation or signal classification files are generated in this mode.

### Mode 2: `with_analysis=True` (Models + Full Analysis) — Default

In this mode, the function iterates over each time period chronologically and calls `train_new_model_for_period` for
each one. This produces the **complete set of outputs** described above for every period:

1. **BERTrend model file:** `{models_path}/bertrend.dill` — updated after each period.
2. **Per-period BERTopic model directories:** `{models_path}/{YYYY-MM-DD}/` — one per period.
3. **Signal classification Parquet files:** `{models_path}/interpretation/{YYYY-MM-DD}/noise.parquet`,
   `weak_signals.parquet`, `strong_signals.parquet` — for each period (starting from the second period, since at least
   two models are needed for signal classification).
4. **LLM interpretation JSONL files:** `{models_path}/interpretation/{YYYY-MM-DD}/noise_interpretation.jsonl`,
   `weak_signals_interpretation.jsonl`, `strong_signals_interpretation.jsonl` — for each period with classified signals.

> **Note:** Signal classification and LLM interpretation are only produced when at least 2 time periods have been
> processed (i.e., `len(bertrend.doc_groups) >= 2`), because topic merging and popularity tracking require comparing
> across periods.

### Optional `since` Parameter

When the `since` parameter is provided, only data with `timestamp >= since` is used. This reduces the number of time
periods processed and consequently the number of output directories created.

## Complete Directory Tree Example

Below is an example of the full output structure after running `regenerate_models` with `with_analysis=True` for a model
with 3 time periods:

```
~/.bertrend/models/users/{user}/{model_id}/
├── bertrend.dill                          # Serialized BERTrend object (cumulative)
├── 2025-01-01/                            # BERTopic model for period 1
│   ├── model files (safetensors)
│   ├── doc_info_df.pkl
│   └── topic_info_df.pkl
├── 2025-01-08/                            # BERTopic model for period 2
│   ├── model files (safetensors)
│   ├── doc_info_df.pkl
│   └── topic_info_df.pkl
├── 2025-01-15/                            # BERTopic model for period 3
│   ├── model files (safetensors)
│   ├── doc_info_df.pkl
│   └── topic_info_df.pkl
└── interpretation/
    ├── 2025-01-08/                        # Analysis for period 2
    │   ├── noise.parquet
    │   ├── noise_interpretation.jsonl
    │   ├── weak_signals.parquet
    │   ├── weak_signals_interpretation.jsonl
    │   ├── strong_signals.parquet
    │   └── strong_signals_interpretation.jsonl
    └── 2025-01-15/                        # Analysis for period 3
        ├── noise.parquet
        ├── noise_interpretation.jsonl
        ├── weak_signals.parquet
        ├── weak_signals_interpretation.jsonl
        ├── strong_signals.parquet
        └── strong_signals_interpretation.jsonl
```

> **Note:** Only non-empty signal categories produce output files. For example, if there are no weak signals for a given
> period, `weak_signals.parquet` and `weak_signals_interpretation.jsonl` will not be created for that period.

## Related Entry Points

The `process_new_data.py` script exposes two CLI commands via [Typer](https://typer.tiangolo.com/):

- **`train-new-model`**: Calls `train_new_model()` → `train_new_model_for_period()` for the latest period only (
  incremental).
- **`regenerate`**: Calls `regenerate_models()` to rebuild all models from scratch.

Usage:

```bash
python process_new_data.py train-new-model <user_name> <model_id>
python process_new_data.py regenerate <user> <model_id> [--with-analysis/--no-with-analysis] [--since YYYY-MM-DD]
```
